# GastroConnect: публикация версии v1003

Эта версия превращает сайт в MVP под 3 роли:

- заведения публикуют смены, смотрят работников, получают отклики и отправляют заявки поставщикам;
- работники заполняют профиль, смотрят смены, откликаются и принимают/отклоняют приглашения;
- поставщики публикуют предложения и видят входящие заявки от заведений.
- у заведения и поставщика есть отдельные формы профиля в кабинете.
- поставщики отвечают на общие запросы заведений, а заведения принимают или отклоняют эти ответы.

Важно: live-сайт сейчас берёт файлы из GitHub Pages. Если GitHub main не обновлён, Cloudflare продолжит отдавать старый `index.html` и старый `supabase.js`, даже когда локальная папка уже исправлена.

## 1. Загрузить файлы в GitHub

Открой репозиторий `alikhanabakarov-cmyk/gastroconnect` и загрузи в корень только эти файлы:

```text
index.html
workers.html
restaurants.html
suppliers.html
auth.html
cabinet.html
admin.html
style.css
script.js
cabinet.js
supabase.js
supabase-marketplace.sql
supabase-full-schema.sql
```

Commit message:

```text
Publish GastroConnect MVP v1003
```

Не загружай служебные папки, архивы, профили браузера и временные файлы.
Если загружаешь готовую папку `publish-v900`, перетаскивай в GitHub именно её содержимое, а не саму папку.

## 2. Проверить, что GitHub Pages обновился

Открой:

```text
https://gastroconnect.ru/index.html?v=1003
https://gastroconnect.ru/supabase.js?v=1003
```

В `supabase.js?v=1003` должно быть:

```js
const SUPABASE_ANON_KEY = 'sb_publishable_0F-CDySnOiJYUdAr8khcJA_QiZc6J2y';
```

Если там старый JWT из трех строк через `.join('.')`, значит файл не обновился.

## 3. Supabase

Таблицы, которые использует фронт:

```text
profiles
worker_profiles
restaurant_profiles
supplier_profiles
shift_posts
shift_applications
shift_invites
supplier_offers
supplier_inquiries
supply_requests
supplier_responses
```

Для общей ленты запросов заведений поставщикам нужно выполнить SQL из:

```text
supabase-marketplace.sql
```

Без этой миграции `supply_requests` будет недоступна, и такие запросы сохранятся только локально в браузере.
Этот же SQL создаёт `supplier_responses` для ответов поставщиков на общие запросы заведений.

Если Supabase проект пустой или таблицы/политики сломаны, вместо точечной миграции используй полный файл:

```text
supabase-full-schema.sql
```

Его нужно выполнять осознанно: он создаёт таблицы, RLS-политики и trigger для профиля нового пользователя.

## 4. Финальная проверка

После загрузки и SQL:

1. Зарегистрируй работника и заполни профиль.
2. Зарегистрируй заведение, создай смену и нажми “Показать работников”.
3. Войди работником, нажми “Показать смены” и отправь отклик.
4. Войди заведением, нажми “Отклики работников”.
5. Зарегистрируй поставщика, опубликуй предложение.
6. Войди заведением, нажми “Показать предложения поставщиков” и отправь запрос.
7. Войди поставщиком, нажми “Показать заявки”.
8. В кабинете заведения проверь кнопки “Принять” и “Отклонить” у откликов.
9. В кабинете поставщика проверь кнопки “Принять” и “Отклонить” у входящих заявок.
10. Войди поставщиком, откликнись на общий запрос заведения, затем войди заведением и проверь “Отклики поставщиков на запросы”.
